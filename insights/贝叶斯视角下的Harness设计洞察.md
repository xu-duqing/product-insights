# 贝叶斯视角下的 Harness 设计洞察

> 用「先验-证据-后验」的方式，重新理解长跑 Agent 的 harness 设计。

## 1. 问题：为什么单 Agent 长跑会失控？

### 1.1 原始先验：

H₀：一个足够强的大模型 + 长上下文 + 少量 prompt，就能高质量完成复杂长跑任务（多小时全栈开发）。

这个先验背后的信念：
- 模型上下文够大，就能记住所有约束；
- 模型能力够强，就能在一个「超级会话」里从 0 跑到 1；
- 只要在 prompt 里反复强调「自检、自评、再思考」，质量就能保证。

### 1.2 真实证据：

随着长跑实验增多，我们看到了一致的 evidence E：

- E1：context anxiety
  - 对话变长后，模型开始莫名提前「收尾」、总结、强行 declare done。
- E2：长任务后期质量显著下滑
  - 任务跑偏、遗忘初衷、逻辑自相矛盾。
- E3：自评严重偏乐观
  - 代码/设计明显有问题，模型自检时仍给出很高评价。
- E4：简单「再想想」式指令，无法修复上述问题
  - 多轮自迭代容易进入自我安慰循环，而不是真正纠错。

从贝叶斯角度：
- 在 H₀ 为真时出现这些现象的概率 P(E | H₀) 较低；
- 在「单 Agent 长跑不够用，需要额外结构」的反假设 H₁ 下，P(E | H₁) 很高。

因此在见到这些 evidence 后，对 H₀ 的后验信念显著下降，我们开始转向 H₁：**复杂长跑任务需要 harness 结构。**

---

## 2. harness = 把「人类工程实践」编码成结构性先验

### 2.1 新先验：多角色、多阶段、多工件

在 H₁ 下，我们假设：

- 长跑任务不是单角色一口气跑完，而是：
  - 规划（Planner）：做产品/架构层的拆解；
  - 生成（Generator）：在约束下扩展细节；
  - 评估（Evaluator）：基于标准做 QA / code review。
- 上下文不靠「记住一切」，而靠**结构化工件**来传递：
  - spec、sprint contract、设计语言、bug list 等。
- 质量保证不靠「自嗨式自评」，而靠**独立 evaluator 的严格反馈闭环**。

本质上，harness 把现实世界中成熟的软件工程模式，当成先验结构 encode 给模型。

> harness 不是在给模型「加轮子」，而是在给整个系统注入一个更符合人类工程实践的结构性先验。

### 2.2 multi-agent 作为能力分布的先验

原始世界观：
- 「一个足够强的 Agent 可以扮演所有角色」。

更新后的世界观：
- 能力在不同角色上分布不同：
  - Planner 更擅长抽象、列举场景和约束；
  - Generator 更擅长在具体约束下展开实现；
  - Evaluator 更擅长在明确标准下发现偏差和问题。

这也是一种先验：我们不再假设单 Agent 能在所有任务上都接近最优，而是假设通过角色拆分，可以更好利用同一模型在不同思维模式下的优势。

---

## 3. Evaluator 的反馈 = 证据，Generator = 基于证据更新

### 3.1 前端设计：把「美不美」做成可观测证据

前端设计是一个高度主观的任务。默认情况下，模型容易滑向「功能可用但审美平庸」的安全区域。

为了解决这个问题，文章里设计了四个评分维度（同时给到 Generator 和 Evaluator）：

1. **Design quality**：整体感与气质——是否是一个统一的作品，而不是组件堆砌；
2. **Originality**：原创性——是否能看出刻意的创作决策，而不是模板+默认样式；
3. **Craft**：工艺——排版层级、间距、色彩和谐度、对比等基本功；
4. **Functionality**：可用性——用户是否容易理解信息架构和主路径。

并且刻意提高对 Design quality 和 Originality 的权重，压低 Craft 和 Functionality 的权重，因为后两者对模型而言相对「默认就不错」，真正需要拉高的是前两者。

在这个 setup 下：

- 这些 criteria 定义了「在好设计假设 H_good 下，Evaluator 期望看到的 observable 特征」。
- Evaluator 对当前设计 x 打分的过程，就是在估计「score(x) 在 H_good 下的似然 P(score | H_good)」。

Generator 的行为是逐步贝叶斯更新：

- 每一轮设计输出 xₜ 被 Evaluator 打分，形成 evidence Eₜ；
- Generator 观察到「什么样的设计模式获得高分、什么样的被惩罚」，相当于更新了对 H_good 下高似然区域的认识；
- 下一轮输出 xₜ₊₁ 朝更高似然区域移动。

> front-end harness 实际上是在用评分标准，把原本模糊的「美不美」转换成一系列可观测的 evidence，从而让生成过程可以像贝叶斯更新那样迭代收敛。

### 3.2 全栈 harness：sprint contract = 定义似然的「公式」

在全栈应用开发的 harness 里，Planner 只给出高层产品 spec，不提前钉死实现细节。每个 sprint 开始前，Generator 和 Evaluator 会协作写一个 **sprint contract**，定义：

- 本轮要交付的具体功能；
- 这些功能的「可观测行为」；
- Evaluator 将如何测试这些行为（通过 UI 流程、API 调用、DB 状态等）。

这个 contract 本质上是：

> 在「应用实现良好」这一假设 H_good 下，我们期望看到的一系列观测结果 {Eᵢ}。

当 Evaluator 执行 contract 中的测试时：

- 如果大多数测试通过，则当前实现的 evidence 与 H_good 相容，P(E | H_good) 较高；
- 如果大量测试失败，则 P(E | H_good) 显著降低，我们对「当前实现已足够好」的后验信念被下调。

Generator 在下一轮改动时，其实就是在用这些失败/通过记录，更新自己对「什么样的实现更可能通过所有验证」的认识，向更高 P(E | H_good) 的方向调整。

> 每一个 sprint 的 QA 过程，都是针对「当前实现是否足够好」这一假设的贝叶斯更新；contract 和评估标准，则是我们事先写好的「似然模型」。

## 4. 模型升级 = 重新校准先验与似然

文章的后半段讲的是：随着模型从 Opus 4.5 升级到 4.6，很多之前看似必须的 harness 结构，开始变得可以简化甚至删掉。

从贝叶斯视角，这可以理解为：

- 在旧模型下：
  - 先验：大模型在长上下文下容易出现 context anxiety；
  - 证据：大量长跑实验证明了这一点；
  - 后验：我们高度相信「必须使用 context reset + handoff artifact」这个工程结构。

- 在新模型下：
  - 新 evidence：4.6 在长上下文下的行为更加稳定，不再明显提前收尾；
  - P(长跑崩溃 | 沿用旧先验) 下降；
  - 后验更新：我们可以回退到更简单的设计——只用单 session + 自动 compaction，而不必强制 reset。

这意味着：

> harness 不是一劳永逸的「架构真理」，而是一套随着新模型 evidence 持续更新的工程假设集合。

我们需要对 harness 做的是：

- 定期重新评估每个组件对输出质量的边际贡献：
  - P(高质量输出 | 有组件 C) vs P(高质量输出 | 无组件 C)；
- 把那些在新模型下「边际贡献很小、成本很高」的组件砍掉；
- 只保留在当前 evidence 下仍显著提高输出质量的 load-bearing 结构。

## 5. 三条「贝叶斯式」工程原则

结合以上，可以把 harness 设计提炼成三条可以指导实战的原则：

### 5.1 先写清「结构性先验」，再搭 harness

在开设计图之前，先问清楚：

- 模型在这个任务上的强项 / 弱项分别是什么？
- 我们愿意假设它在哪些场景能「一口气跑完」，在哪些场景需要人工分段？
- 哪些人类工程实践（多角色、多阶段、多工件）值得被 encode 成先验结构？

只有在这些假设写清后，harness 的多 Agent 结构才有依据，而不是凭感觉堆组件。

### 5.2 把「好坏」变成结构化 evidence，而不是模糊感觉

无论是前端设计还是全栈开发：

- 不要满足于「感觉还不错」这种顽皮评价；
- 把你对「好设计、好产品、好代码」的直觉，拆成可打分、可验证的 criteria；
- 让 Evaluator 在这些 criteria 上产出结构化信号：分数、点评、bug 列表、未满足的 user story 等。

只有当 evidence 被结构化，Generator 才有足够信息去做类似贝叶斯更新，而不是在黑箱里瞎试。

### 5.3 持续根据新 evidence 简化 harness，而不是迷信复杂度

每个 harness 组件，都隐含着一个命题：「模型自己做不到 X」。

当模型升级、任务场景变化、新实验结果出现时：

- 这条命题的后验需要被重新估计；
- 对那些「现在模型已经能自己搞定」的部分，要敢于删掉 scaffold；
- 保持一个健康的「怀疑态度」：
  - 复杂度要为 clear 的性能提升买单；
  - 提升不再明显时，就应该减负。

> 用贝叶斯的话说：harness 的设计不是一次定型，而是在不断引入新 evidence、重估先验、更新后验的过程。真正成熟的工程实践，是在这个循环里不断做减法。

## 6. 对产品思考的启发

从产品角度看，这篇 harness 实验背后有两层更通用的启发：

1. **产品结构 = 你对用户/系统的先验**
   - 你怎么切页面、怎么安排流程、怎么划分权限，其实都在 encode 你对「用户会怎么用」「系统会怎么崩」的信念；
   - 当数据和反馈不断进来时，应该像更新贝叶斯后验一样，逐步修改这些结构性假设，而不是固守第一版 IA。

2. **评估体系不是事后补充，而是前置的「似然模型」设计**
   - 如果你期望一个系统往某个方向进化（更高留存、更强 engagement、更优转化），就要提前设计好「什么样的行为被奖励、什么样被惩罚」；
   - 对 AI 系统来说，这就是 prompt+criteria+evaluator；
   - 对传统产品来说，就是指标体系、激励机制、运营节奏——它们决定了系统在 evidence 下如何更新自身状态。

把 harness 放在贝叶斯框架下看，它不再只是「多加几个 Agent」的工程技巧，而是一套围绕先验、证据和更新逻辑的系统设计方法论。
